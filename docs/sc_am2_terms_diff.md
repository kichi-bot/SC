# SC午前Ⅱ 混同しやすい用語の違い

> 午前Ⅱのひっかけ選択肢は、ほぼこのページの「混同」から作られる。1項目30秒で確認できる形にしてある。

## SPF / DKIM / DMARC

- **SPF**:送信元**IPアドレス**を検証。ドメインのDNS(TXT)に正規送信サーバのIPを登録し、受信側がエンベロープFromのドメインと照合
- **DKIM**:**デジタル署名**を検証。送信側サーバが署名を付与、受信側がDNSの公開鍵で検証。転送に強い
- **DMARC**:SPF/DKIM**失敗時の処理方針**(none/quarantine/reject)を送信側ドメイン管理者が指定+レポート
- 覚え方:「SPFはIP、DKIMは署名、DMARCはポリシー」
- ひっかけ:SPFがSubjectやヘッダーFromを見る、という選択肢は誤り

## IDS / IPS / WAF

- **IDS**:不正通信を**検知して通知**(遮断しない)。ミラーポート設置が典型
- **IPS**:通信経路上(**インライン**)に置き、検知したら**遮断**まで行う
- **WAF**:**Webアプリケーション層**(HTTP)に特化して攻撃パターンを検知・遮断
- 方式の軸も混同注意:シグネチャ型=既知パターン照合/アノマリ型=正常状態からの逸脱検知
- ひっかけ:フォールスポジティブ=正常通信を攻撃と誤検知(WAF問題の定番)

## ハッシュ / MAC / デジタル署名

- **ハッシュ**:鍵なし。**完全性**(改ざん検知)のみ
- **MAC**:**共通鍵**+メッセージ。完全性+**通信相手の確認**。ただし鍵を双方が持つので第三者への証明(否認防止)は不可
- **デジタル署名**:**秘密鍵**で署名・公開鍵で検証。完全性+**否認防止**(第三者に証明できる)
- 覚え方:「鍵なし→共通鍵→公開鍵」の順に保証が強くなる

## 共通鍵暗号 / 公開鍵暗号

- **共通鍵**:暗号化と復号が**同じ鍵**。高速。鍵配送が課題。n人で n(n-1)/2 個
- **公開鍵**:**鍵ペア**。暗号化は「受信者の公開鍵」、署名は「送信者の秘密鍵」。低速
- 実運用は**ハイブリッド**:データは共通鍵で、その共通鍵を公開鍵で受け渡す
- ひっかけ:「誰の」「どちらの」鍵かを入れ替えた選択肢

## CRL / OCSP

- **CRL**:失効した証明書のシリアル番号**一覧をCAが定期発行**。受信側でリストを照合
- **OCSP**:失効状態を**リアルタイムに問い合わせ**る。応答するのはVA(OCSPレスポンダ)
- 共通の注意:載るのは「**有効期限内に**失効」した証明書。期限切れは対象外
- ひっかけ:「CRLには期限切れ証明書が登録される」は誤り

## SAML / OAuth

- **SAML**:**認証**の連携(SSO)。IdPが認証→**アサーション**発行→SPが検証。OASIS策定
- **OAuth 2.0**:**認可**の委譲。利用者の同意の下、別サービスに**アクセストークン**で限定アクセスを許す
- 覚え方:「SAML=あなたは誰か(認証)、OAuth=何をしてよいか(認可)」
- ひっかけ:OAuthを認証プロトコルとする選択肢(認証はOpenID Connectの役割)

## XSS / CSRF / SQLインジェクション

- **XSS**:被害者の**ブラウザで**スクリプトが実行される。対策=出力時エスケープ、CSP
- **CSRF**:ログイン済み利用者に**意図しないリクエストを送信させる**。対策=秘密トークン、再認証、SameSite
- **SQLi**:入力値で**SQL文を改変**しDBを操作。対策=プレースホルダ(バインド機構)
- ひっかけ:CSRF対策として「エスケープ処理」を選ばせる(それはXSS対策)——実際に3回出題

## TCP / UDP

- **TCP**:コネクション型(3ウェイハンドシェイク)。信頼性あり。HTTP/SMTPなど
- **UDP**:コネクションレス。高速・軽量。DNS/NTP/音声など。**送信元偽装が容易→リフレクション攻撃に悪用**
- ヘッダーの中身:ポート番号は**TCP/UDPヘッダー**、IPアドレスは**IPヘッダー**(出題実績3回)
- 派生:DTLS=UDP上でTLS相当の保護

## DoS / DDoS(と反射型)

- **DoS**:単一発信元からの妨害攻撃
- **DDoS**:**多数の踏み台(ボットネット)**からの分散攻撃
- **反射型(DRDoS)**:送信元を**標的に偽装**してサーバ群に問合せ→**応答が標的に集中**。DNS/NTPの増幅を悪用
- 個別手口:Smurf=ICMPエコー要求をブロードキャストへ/ICMP Flood=大量ping/バックスキャッタ=偽装された側にSYN/ACKが届く現象

## CVE / CVSS(+CWE)

- **CVE**:個々の脆弱性の**識別子**(例 CVE-2024-XXXX)。MITREが事務局
- **CVSS**:脆弱性の**深刻度を数値評価**する共通基準(0.0〜10.0)
- **CWE**:脆弱性の**種類**の一覧(例 XSSはCWE-79)
- 覚え方:「CVE=名前、CVSS=点数、CWE=種類」——3つの混同がひっかけの定番

## 認証 / 認可

- **認証(Authentication)**:相手が**本人か**を確認する(パスワード、生体、証明書)
- **認可(Authorization)**:認証済みの主体に**何を許すか**を決める(アクセス制御、権限)
- +**Accounting**(利用記録)でAAA(RADIUS/Diameterが提供。R6秋出題)
- 午後では「認証は通るが認可チェックがない」IDOR脆弱性として頻出

## 暗号化 / ハッシュ化 / 符号化

- **暗号化**:鍵があれば**元に戻せる**(機密性のため)
- **ハッシュ化**:**元に戻せない**一方向変換(完全性・パスワード保存のため)
- **符号化(エンコード)**:表現形式の変換。**誰でも戻せる**(Base64等。セキュリティ対策ではない)
- ひっかけ:「Base64で暗号化」という表現は常に誤り

## おまけ:直近試験で問われた新しめの対比

- **CASB**(クラウド利用の可視化) / **CSPM**(クラウド設定の監視) / **SIEM**(ログ相関分析) / **SOAR**(対応自動化) / **UEBA**(振る舞い検知)
- **CSIRT**(自組織のインシデント対応) / **PSIRT**(自社**製品**の脆弱性対応。R7秋出題) / **SOC**(監視・分析の実働部隊)
- **量子暗号(QKD)**(盗聴を検知できる鍵配送) / **PQC**(量子計算機に耐える従来型暗号)
- **サイバーキルチェーン**(攻撃の段階モデル) / **MITRE ATT&CK**(戦術・技術のナレッジベース)
