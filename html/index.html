01. 過去問ストーリー一覧 事実

この章の要点
  • R3春〜R7秋の午後問題を、業務ストーリー単位で一覧化した「実際に出た事実」の整理。
  • 試験形式はR5秋から 午後Ⅰ＋午後Ⅱ → 統合午後(4問中2問選択) に変更。
  • 大分類の登場頻度：Web=毎回級、認証・認可/インシデント=高頻度、委託・サプライチェーン=急増中。
  • 各表は横スクロール可。難易度は採点講評の正答率表現を根拠にした相対評価。

特に頻出・高配点の失点パターン深掘り

◆ P5 暗号技術の混同 —「デジタル署名」と書けば当たる、ではない
  • R7秋 問2 設問3(2)：改ざん検知の手段を問われ「デジタル署名」で失点。正答はMAC(メッセージ認証符号)。共通鍵を共有する当事者間の完全性保証にはMACが適切。
  • R5秋 問3 設問3(3)：HSMの利点を「電子署名を暗号化できる」「秘密鍵が漏れても安全」と誤解。正答は「秘密鍵がHSMの外に出ない/漏れない」。
教訓：暗号は「守りたい性質」から逆算する。機密性→暗号化、完全性+共通鍵保持者→MAC、完全性+第三者に証明→デジタル署名。
◆ P6 SPF/DKIM/DMARCの判定対象
  • R6秋 問2 設問5：SPF側で「Subjectに通番」→ SPFはSubjectを判定対象にしないので誤り。SPFはエンベロープFromのドメインと送信元IPを照合。
  • DKIM側では「hタグにSubjectが含まれる」ため、Subjectを変更するとDKIM署名検証が失敗する、という因果を答える必要があった。
教訓：SPF=IPアドレス、DKIM=電子署名(hタグで対象指定)、DMARC=結果を使ったポリシー+レポート。判定材料を混ぜない。
◆ P3 攻撃原因と対策の非対応 —「効かない対策」を書く典型
  • R7秋 問4 設問3：マルウェア対策として「SCADAの利用者認証強化」。講評は「マルウェア対策として利用者認証の強化はあまり効果がない」と明記。
  • R7秋 問1 設問4：脆弱性を作り込まない開発プロセスを問うたのに、「ログ分析」「ID棚卸」「WAF導入」など運用対策・製品導入を解答。
教訓：対策を書く前に「この攻撃はどの段階で成立するか」を特定し、その段階を塞ぐ対策だけを選ぶ。
◆ P2 理由不足 — SBOMは「定義」でなく「使い方」
  • R7春 問1 設問4：SBOM(ソフト部品表)の利点を問われ定義を書いただけで失点。正答は「利用ソフトとバージョンが明確になり、脆弱性の影響有無を容易に把握できるから」。
◆ P9 攻撃連鎖 — 単発脆弱性ではなく"つなげて"悪用する
  • R6春 問3：XSS検出箇所(管理者のみ実行可)と利用者情報取得(会員管理機能)を結びつけて攻撃を構成。CSRFではトークンを攻撃者が自分のアカウントで取得して悪用する部分が書けず失点。
  • R6秋 問4 設問2g：複数脆弱性を組み合わせた攻撃の「後の流れ」に合わない解答が散見。
教訓：近年は「脆弱性A×機能B×脆弱性C」の連鎖問題が増加。攻撃者視点で時系列に物語化する。

この章の要点

1. Webアプリ脆弱性（XSS / SQLi / CSRF / SSRF）
XSS（他人のブラウザで悪意のスクリプトを実行させる攻撃）
  • 格納型：スクリプトがサーバに保存され閲覧者全員で実行(掲示板等)。被害大
  • 反射型：URLパラメータ等のスクリプトがその場で返り実行。リンクを踏ませる
  • DOMベース：サーバを介さずブラウザ内JSがDOMを書換え実行。※DOMを使う=DOMベースではない
対策：□(出力箇所)にデータを出力する前にエスケープ処理を施す。
緩和：CSP(読み込み元を制限)を設定し、許可した配信元以外のスクリプト実行を禁止する。
cookie保護：cookieにHttpOnly属性を付与しスクリプトからの読み取りを防ぐ／別ドメインのURLへcookieが送られない仕組みにする。
SQLインジェクション
対策：□の処理を、プレースホルダを用いたPreparedStatementで実装する。※Statement(文字列連結)ではダメ
CSRF（利用者に意図しない操作を送信させる攻撃）
攻撃：攻撃者が用意したわなフォームに利用者を誘導し、利用者の権限で□を実行させる。
対策：リクエストにCSRFトークンを付与し、サーバ側で照合する。
SSRF（サーバに攻撃者指定の宛先へ通信させる攻撃）
攻撃：パラメータ□の値をIMDS(クラウドのメタデータ提供機能)のURLに変更し、クレデンシャルを取得する。
対策：パラメータ□の値が許可した□以外のURLならエラーにする。
Webヘッダ系
  • HSTS：□への接続をHTTPSに強制する
  • X-Content-Type-Options：Content-Typeで指定したMIMEタイプを強制適用する
  • クリックジャッキング対策：X-Frame-Options / CSPのframe-ancestorsでフレーム埋め込みを制限する
2. 認証 / 認可 / API（JWT・OAuth・OIDC）
認証(Authentication)認可(Authorization)
問い「あなたは誰か」「あなたは何をしてよいか」
例ログイン、MFAロール、アクセス制御、スコープ
崩れるとなりすまし他人のデータにアクセス(IDOR)
認可不備(IDOR)対策：リクエストの利用者IDと□(資源)の所有者が一致するかをサーバ側で検証する。
推測困難化：□(ID)を推測困難な値になるよう生成方法を変更する。
JWT（署名付きトークン）
alg=none対策：JWTヘッダのalgの値がNONEでないことを検証する。
改ざん対策：JWTに含まれる利用者IDが□(セッションの利用者)と一致するか検証する。
OAuth 2.0 / OpenID Connect
  • OAuth=認可(アクセス権の委譲)、OIDC=OAuthの上で認証を行う仕組み
  • PKCE(認可コード横取り対策)：認可時にcode_verifierのハッシュを送り、トークン交換時にcode_verifierを検証
  • redirect_uri：登録済みと完全一致する場合のみ許可／state・nonce：CSRF・リプレイ対策
3. マルウェア / ログ / インシデント対応（WAF・IDS・IPS・EDR・DLP・SIEM）

4. 暗号 / 鍵 / 証明書（署名・MAC・ハッシュ・暗号化 / PKI）
守りたい性質使う技術補足
機密性(見られない)暗号化(共通鍵/公開鍵)AES等
完全性＋当事者間の真正性MAC(メッセージ認証符号)共通鍵を共有する相手同士

証明書・PKI（サーバ証明書の検証項目）
  1. 信頼された認証局(CA)が発行したものか
  2. 証明書のサーバ名(CN/SAN)が接続先と一致するか
  3. 有効期限内か
  4. 失効していないか（OCSP=オンライン失効確認／CRL=失効リスト）

5. リモートワーク / VPN / 多要素認証（FIDO・中間者）
中間者型フィッシング手順：攻撃者が偽サイトに入力された利用者ID/パスワードを正規サイトへ中継し、正規利用者に送られたMFAコードを偽サイトで受け取って正規サイトに入力し、認証を突破する。
フィッシング耐性：FIDO/WebAuthn(パスキー)は、認証情報にオリジン(接続先)と署名を含めサーバが検証するため、偽サイトでは認証が成立せずフィッシング耐性がある。
入力先の限定：認証情報は、メール内リンクから開いた画面には入力せず、正規の□(VPNダイアログ等)にだけ入力する。
接続元制限：□サイトへのアクセスを許可する接続元IPアドレスを□に制限する。

MFA代替(故障時)：セキュリティ強度を落とさない代替(別のFIDO認証器登録、バックアップコードの安全管理等)を認証基盤の仕様に沿って書く

12. 混同ペア・最終確認カード（試験直前に音読）
  • 認証（誰か）↔ 認可（何してよいか）
  • デジタル署名（第三者に証明・公開鍵）↔ MAC（当事者間・共通鍵）↔ ハッシュ（鍵なし）↔ 暗号化（機密性）

  • SAST（静的）↔ DAST（動的）↔ IAST（対話型）
