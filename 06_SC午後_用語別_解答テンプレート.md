# 06_SC午後_用語別_解答テンプレート

> **目的**：午後問題で「そのまま骨格として使える解答文」と「混同しやすい用語の区別」を一冊に。
> **使い方**：□には本文の固有名詞(システム名・機能名・パラメータ名・IPアドレス)を入れる。テンプレは"型"であり、本文条件への当てはめ(`03`のP1対策)を必ず行うこと。
> **初学者向け**：難しい用語には（かっこ書き）で短い説明を付けています。

---

## 1. Webアプリ脆弱性（SC-ST-001）

### XSS（クロスサイトスクリプティング＝他人のブラウザで悪意のスクリプトを実行させる攻撃）
- **3類型の区別**：
  - **格納型**：スクリプトがサーバに保存され、閲覧者全員で実行される（掲示板・レビュー等）。被害大
  - **反射型**：URLパラメータ等のスクリプトがその場で返って実行される。攻撃はリンクを踏ませる
  - **DOMベース**：サーバを介さず、ブラウザ内のJavaScriptがDOMを書き換えて実行。※DOMを使う＝DOMベース、ではない（R5秋の頻出誤答）
- **対策テンプレ**：「□(出力箇所)にデータを出力する前にエスケープ処理を施す。」
- **緩和テンプレ**：「CSP(Content Security Policy＝読み込み元を制限する仕組み)を設定し、許可した配信元以外のスクリプト実行を禁止する。」
- **cookie保護**：「cookieにHttpOnly属性を付与し、スクリプトからの読み取りを防ぐ。」/「スクリプトから別ドメインのURLへcookieが送られない仕組みにする。」

### SQLインジェクション
- **対策テンプレ**：「□の処理を、プレースホルダを用いたPreparedStatementで実装する。」
  - ※Statement(文字列連結)ではダメ。PreparedStatement＝値を後から安全にはめ込む仕組み（R4春・R5春の頻出誤答）
- **保険的対策**：「入力値を想定した文字種・長さに制限し、抽出条件に□を追加する。」

### CSRF（クロスサイトリクエストフォージェリ＝利用者に意図しない操作を送信させる攻撃）
- **攻撃テンプレ**：「攻撃者が用意したわなフォームに利用者を誘導し、利用者の権限で□(操作)を実行させる。」
- **対策テンプレ**：「リクエストにCSRFトークンを付与し、サーバ側で照合する。」

### SSRF（サーバサイドリクエストフォージェリ＝サーバに攻撃者指定の宛先へ通信させる攻撃）
- **攻撃テンプレ**：「パラメータ□の値をIMDS(クラウドのメタデータ提供機能)のURLに変更し、クレデンシャル(認証情報)を取得する。」
- **対策テンプレ**：「パラメータ□の値が許可した□以外のURLならエラーにする。」

### その他Webヘッダ系
- **HSTS**：「□への接続をHTTPSに強制する。」
- **X-Content-Type-Options**：「Content-Typeで指定したMIMEタイプを強制適用する。」
- **クリックジャッキング対策**：「X-Frame-Options / CSPのframe-ancestorsでフレーム埋め込みを制限する。」

## 2. 認証・認可・API（SC-ST-002）

### 認証 vs 認可（最重要の区別）
| | 認証(Authentication) | 認可(Authorization) |
|---|---|---|
| 問い | 「あなたは誰か」 | 「あなたは何をしてよいか」 |
| 例 | ログイン、MFA | ロール、アクセス制御、スコープ |
| 崩れると | なりすまし | 他人のデータにアクセス(IDOR) |

- **認可不備(IDOR/BOLA)対策テンプレ**：「リクエストの利用者IDと□(資源)の所有者が一致するかをサーバ側で検証する。」
- **推測困難化テンプレ**：「□(ID)を推測困難な値になるよう生成方法を変更する。」

### JWT（JSON Web Token＝署名付きのトークン）
- **alg=none攻撃対策**：「JWTヘッダのalgの値がNONEでないことを検証する。」
- **改ざん対策**：「JWTに含まれる利用者IDが□(セッションの利用者)と一致するか検証する。」

### OAuth 2.0 / OpenID Connect
- **OAuth**：認可の仕組み（アクセス権の委譲）。**OIDC**：OAuthの上で認証を行う仕組み
- **PKCE**（認可コード横取り対策）：「認可リクエスト時にcode_verifierのハッシュ(code_challenge)を送り、トークン交換時にcode_verifierを検証する。」
- **redirect_uri**：「登録済みリダイレクトURIと完全一致する場合のみ許可する。」
- **state/nonce**：CSRF/リプレイ対策の値

## 3. マルウェア・ログ・インシデント対応（SC-ST-004）

### ログ調査テンプレ
- **検出テンプレ**：「□のログから、□(条件)に該当する□(ホスト/利用者)を検出する。」
  - 例：「マルウェア対策サービスのログから、対策ソフトが停止したホストを検出する。」
- **調査の観点**：プロセスの**親子関係**（不審プロセスがどこから起動されたか）、通信先URL、タスク登録、認証(RDP)ログ、タイムライン整理

### 対策製品の役割（混同注意）
| 製品 | 守る層 | 役割 |
|---|---|---|
| **WAF** | Web境界 | Webアプリへの攻撃を検知・遮断 |
| **IDS/IPS** | ネットワーク | 侵入を検知(IDS)/防御(IPS) |
| **EDR** | エンドポイント | 端末の不審挙動を検知し対応(隔離等) |
| **DLP** | データ | 情報の持ち出しを検知・制限 |
| **SIEM** | 横断 | 各種ログを相関分析 |

- **持ち出し対策テンプレ**：「□にDLPを導入し、ファイルの持ち出しを制限する。」（※「アクセス権最小化」では持ち出しは防げない＝R6秋の誤答）
- **横展開対策**：「仮想PC/サーバへのRDP接続に対し、接続元IPアドレスによる制限を行う。」

## 4. 暗号・鍵・証明書（SC-ST-007）

### 暗号技術の使い分け（最頻出の混同ポイント）
| 守りたい性質 | 使う技術 | 補足 |
|---|---|---|
| 機密性(見られない) | 暗号化(共通鍵/公開鍵) | AES等 |
| 完全性(改ざん検知)＋当事者間の真正性 | **MAC(メッセージ認証符号)** | 共通鍵を共有する相手同士 |
| 完全性＋第三者に送信者を証明 | **デジタル署名** | 公開鍵ペア。否認防止 |
| データの同一性確認 | ハッシュ(SHA-2) | 鍵なし |

- **注意**：改ざん検知＝反射的に「デジタル署名」と書かない。共通鍵共有の文脈なら**MAC**（R7秋の頻出誤答）
- **HSM(Hardware Security Module)の利点テンプレ**：「秘密鍵がHSMの外部に出ないため、鍵の漏えいを防げる。」（※「署名を暗号化できる」等は誤り）

### 証明書・PKI
- **サーバ証明書の検証項目**（列挙できるように）：
  1. 信頼された認証局(CA)が発行したものか
  2. 証明書のサーバ名(CN/SAN)が接続先と一致するか
  3. 有効期限内か
  4. 失効していないか（**OCSP**＝オンラインで失効確認/**CRL**＝失効リスト）
- **偽サイト検知テンプレ**：「HTTPSでアクセスするため、偽サイトのサーバ証明書は検証に失敗する。」
- **無線LAN(EAP-TLS)**：「EAP-TLSに必要な認証情報(秘密鍵)は業務PCから取り出せないようにし、業務PCにしか格納できないようにする。」

## 5. リモートワーク・VPN・多要素認証（SC-ST-008）

- **リアルタイム型(中間者)フィッシングの手順テンプレ**：
  「攻撃者が偽サイトに入力された利用者ID/パスワードを正規サイトへ中継し、正規利用者に送られたMFAコード(ワンタイム)を偽サイトで受け取って正規サイトに入力し、認証を突破する。」
- **フィッシング耐性の説明**：「FIDO/WebAuthn(パスキー)は、認証情報にオリジン(接続先)と署名を含めサーバが検証するため、偽サイトでは認証が成立せずフィッシング耐性がある。」
- **入力先の限定テンプレ**：「認証情報は、メール内のリンクから開いた画面には入力せず、正規の□(VPNダイアログ等)にだけ入力する。」
- **接続元制限**：「□サイトへのアクセスを許可する接続元IPアドレスを□に制限する。」
- **MFA代替(故障時)**：セキュリティ強度を落とさない代替（別のFIDO認証器登録、バックアップコードの安全な管理等）を、認証基盤の仕様に沿って書く

## 6. クラウド・SaaS（SC-ST-006）

- **通信経路**：移行前後で「どこを経由するか」を図で追い、変化点を答える（R5春Ⅰ問3）
- **フィルタリング**：「ルールは適用順序で動作が変わる」ため番号順に注意（R5春Ⅰ問3 設問5）
- **IMDS/SSRF**：4章のSSRFテンプレ参照。IMDSv2はトークン必須で緩和
- **最小権限テンプレ**：「□(利用者/API)に必要最小限の権限だけを付与する。」
- **コンテナ**：権限・仕組みを理解（どの権限で実行されるか）。RBAC、イメージスキャン、Secrets管理

## 7. メールセキュリティ（SC-ST-003）

### SPF / DKIM / DMARC の判定対象（丸暗記）
| 方式 | 何を見るか | 一言 |
|---|---|---|
| **SPF** | 送信元IPアドレス と エンベロープFromのドメイン | Subjectは見ない |
| **DKIM** | 電子署名（hタグで対象ヘッダを指定、Subjectを含むこともある） | 署名対象を変えると検証失敗 |
| **DMARC** | SPF/DKIMの結果＋Fromドメインの整合(アライメント) | ポリシー(none/quarantine/reject)とレポート |

- **SPF設定テンプレ**：「SPFレコードに、□(新サービス)のメール送信元IPアドレスを追加する。」
- **DKIM失敗の因果テンプレ**：「DKIMのhタグにSubjectが含まれるのに、□(ML等)でSubjectが変わるため、署名検証が失敗する。」
- **SMTPS**：SMTP通信自体の暗号化（※S/MIMEは本文/添付の暗号化・署名であって、SMTP暗号化ではない＝R6秋の誤答）

## 8. 委託先・サプライチェーン・脆弱性管理（SC-ST-005 / 009）

### サプライチェーン
- **再委託テンプレ**：「委託先に要求する対策と同等のセキュリティ対策を、再委託先にも要求させる。」
- **セキュリティバイデザイン**：「企画・設計工程からセキュリティ対策を組み込む考え方。」
- **SBOM(ソフトウェア部品表)の利点テンプレ**：「利用しているソフトウェアやバージョンが明確になり、脆弱性の影響有無を容易に把握できるから。」（※定義だけはNG）

### テストツールの区別
| 略語 | 名称 | 特徴 |
|---|---|---|
| **SAST** | 静的解析 | ソースコードを解析。実行不要、早期発見 |
| **DAST** | 動的解析 | 稼働中アプリに攻撃を送り検出 |
| **IAST** | 対話型 | 実行時にコード内部も観測、両者の中間 |

### 脆弱性管理の指標
| 指標 | 何を表すか | 使いどころ |
|---|---|---|
| **CVSS基本値** | 脆弱性そのものの深刻度 | 技術的特性。AC=攻撃条件の複雑さ(※機密性影響Cと混同注意) |
| **CVSS現状値** | 攻撃コードの有無等の"今"の状況 | 公開情報から自分で評価する手間がある |
| **EPSS** | 悪用される確率(予測) | FIRSTの公開値をそのまま使える |
| **KEVカタログ** | 実際に悪用が確認された脆弱性リスト | 「実際に悪用された」→優先度を上げる根拠 |
- **優先度テンプレ**：「KEVカタログに掲載され実際に悪用されているため、対応優先度を上げる。」

### ASM（アタックサーフェスマネジメント）
- **サブドメインテイクオーバー対策**：「終了したサブドメインのCNAMEレコードを(権威DNSサーバから)削除する。」
- **稼働調査**：「ポートスキャンで公開ポートを確認し、脆弱性スキャナーで脆弱性の有無を確認する。」
- **廃止判断**：「継続しない場合は速やかにネットワークから切り離し、機器を廃棄する。」

## 9. セキュアコーディング・開発プロセス（SC-ST-010）

- **PreparedStatement**：3章SQLi参照
- **例外処理(Java)**：「例外発生時でも必ず実行される`finally`でリソースを解放する。」/「捕捉した例外は握りつぶさず再throwする。」
- **パスワード保存**：「SHA-256等のハッシュ(＋ソルト)で保存する。」（可逆な暗号化ではない）
- **最小権限**：DBアクセス権を機能ごとに必要最小限に
- **「開発プロセスの対策」を問われたら**：設計/実装/テスト/レビュー/専門家による脆弱性検査など**プロセス上の施策**に限定。運用対策(ログ分析・棚卸)や製品導入(WAF)を書かない（R7秋の頻出誤答）

## 10. リスクアセスメント・体制（SC-ST-013）

- **リスク対応4分類**：回避／移転／低減(軽減)／保有(受容)
- **攻撃シナリオテンプレ**：「攻撃者が□(手口)により□(初期侵入)し、□(横展開)して、最終的に□(被害)する。」
  - リスクは**具体的**に、**被害の大きさ・発生頻度**まで踏み込む（R5秋の失点対策）
- **権限マトリクス**：所属組織×(読込/書込/権限変更)を最小権限で設計
- **レジリエンス(復旧優先)テンプレ**：「被害発生時に迅速復旧するため、平常時にバックアップを取得しておく。」（※復旧を問われて攻撃阻止を書かない＝R7秋の誤答）

## 11. ネットワーク基盤（SC-ST-011）

- **DNSSEC**：「DNS応答にデジタル署名を付け、改ざん(キャッシュポイズニング)を検知する。」
- **DoH/DoT**：「DNS通信をHTTPS/TLSで暗号化し、盗聴・改ざんを防ぐ。」
- **ゾーン転送制限**：「ゾーン転送要求を許可するホストを必要最小限(□)に制限する。」
- **IEEE802.1X/RADIUS**：認証後にネットワーク接続を許可。EAP-TLSは証明書認証
- **不正DHCP検出**（※不正AP検出と混同注意）：DHCP応答を監視し、許可外のDHCPサーバを検知

## 12. 混同ペア・最終確認カード（試験直前に音読）

- 認証（誰か）↔ 認可（何してよいか）
- デジタル署名（第三者に証明・公開鍵）↔ MAC（当事者間・共通鍵）↔ ハッシュ（鍵なし）↔ 暗号化（機密性）
- SPF（送信元IP）↔ DKIM（署名・hタグ）↔ DMARC（ポリシー）
- WAF（Web境界）↔ IPS（NW侵入）↔ EDR（端末）↔ DLP（データ持出）↔ SIEM（ログ相関）
- 格納型XSS ↔ 反射型XSS ↔ DOMベースXSS
- SAST（静的）↔ DAST（動的）↔ IAST（対話型）
- CVSS基本値（深刻度）↔ 現状値（今の状況）↔ EPSS（悪用確率）↔ KEV（悪用実績）
- Statement（危険）↔ PreparedStatement（安全・プレースホルダ）
- OAuth（認可・権限委譲）↔ OIDC（認証）
- パスワードリスト（漏えいID再利用）↔ スプレー（少数PWを多数IDへ）↔ 総当たり（1IDに全PW）

> 本テンプレは過去問(R3春〜R7秋)の解答例・採点講評から抽出。実戦では必ず本文の固有条件へ当てはめること（`03`のP1〜P12を回避）。
